Enrichment Pipeline

From a state permit row to the answer in an AI search.

Public permit feeds are noisy, unstructured, and invisible to search engines. The J Becher portal runs every record through a five-stage pipeline — classifying scope, enriching with Claude, attaching schema.org structured data, and publishing as a branded, AI-discoverable project page on WordPress. The goal isn't just SEO. It's appearing as the cited contractor when someone asks Perplexity, ChatGPT, Google SGE, or Bing Copilot "who handles multifamily electrical in Hennepin County?"

Stage Walkthrough

What actually happens inside each stage.

1
Raw Permit Ingest
Pulled nightly from the MN state electrical permit feed.
What goes in

A flat permit row keyed by permit ID. Fields include applicant, GC, scope text, site address, county, permit type, fee, and filing date. Free-text fields are inconsistent — "800A METER BANK" in one row, "800-amp service entrance" in the next.

What comes out

A normalized row with canonicalized addresses, parsed scope tokens (unit counts, electrical service amperage, project type), and a clean GC name. Nothing is "enriched" yet — this stage is about making the row machine-readable.

2
Classification — Skip / Light / Full
Triage every row so Claude only sees the records worth narrating.
skip

Single-family service swaps, panel replacements, and generic residential rough-ins. Stored for the dashboard counts but not given a page. Saves token spend on low-signal records.

enrich_light

Mid-size jobs — single-family new construction, small commercial, EV charger installs. Gets a short SEO headline + body, but no full project narrative. Schema markup is still attached.

enrich_full

Multifamily, healthcare, large commercial, marquee GC partnerships. Full Claude treatment: headline, body, ad preview, tags, and complete ConstructionProject schema.

Classification logic (simplified)
// classifier.ts function classify(row) { if (row.fee < 300 && row.unit_count <= 1) return "skip"; if (row.project_type === "res_multi") return "enrich_full"; if (row.project_type === "commercial" && row.fee >= 1500) return "enrich_full"; if (row.scope_tokens.includes("ev_charger")) return "enrich_light"; return "enrich_light"; }
3
AI Enrichment — Claude API
A single Claude call returns a typed, schema-validated copy block.
Why Claude, and why structured output

The model is asked for a strict JSON object with named fields — headline, body, tags, ad_title, ad_desc. Structured output is enforced by a tool-use schema, so the response is never free-form prose that has to be parsed. If the model returns a malformed field, the row is retried with a stricter prompt or routed to manual review.

The prompt includes J Becher's brand voice (union-trained, IBEW Local 292 signatory, Twin Cities multifamily depth) and forbids hallucinated facts — Claude only narrates what's in the row.

Claude output (validated JSON)
{ "headline": "10-Unit Townhome with 800A Service — Frerichs Construction", "body": "J Becher & Associates is providing full electrical scope for a 10-unit townhome project for Frerichs Construction on University Avenue SE in Minneapolis, Hennepin County...", "tags": ["Multifamily", "New Construction", "Service Upgrade", "800A Meter Bank"], "ad_title": "Townhome Electrical & Service Upgrades — Minneapolis | J Becher", "ad_desc": "IBEW Local 292 contractor for 10-unit townhomes, 800A meter banks, and service entrance work." }
4
Schema Markup — JSON-LD
The piece that turns a webpage into a fact the AI can cite.
What schema attaches to each page
LocalBusiness
Identifies J Becher as a real, named business with address, service area, and union affiliation.
Service
Describes the specific service category — multifamily electrical, EV charger install, service upgrade.
ConstructionProject
Wraps the project itself: location, GC, scope, unit count, completion status.

Pages without schema are unreadable to AI search. The crawler may see "10 unit townhome" in the body text, but with no schema it can't confidently link that fact to J Becher as the electrical contractor of record. With schema, the relationship is explicit and machine-resolvable.

Example JSON-LD (rendered in page head)
{ "@context": "https://schema.org", "@type": "ConstructionProject", "name": "10-Unit Townhome — University Ave SE", "location": { "@type": "Place", "address": { "@type": "PostalAddress", "streetAddress": "2929 University Ave SE", "addressLocality": "Minneapolis", "addressRegion": "MN", "postalCode": "55414" } }, "contractor": { "@type": "LocalBusiness", "name": "J Becher & Associates", "memberOf": "IBEW Local 292", "areaServed": "Twin Cities, MN" }, "generalContractor": "Frerichs Construction", "numberOfUnits": 10, "serviceCategory": "Multifamily Electrical" }
5
WordPress Publishing
Rendered as a project page, indexed, and pinged to AI crawlers.

Each enriched record becomes a project page on the J Becher WordPress site. The page includes the Claude-generated body, the project tags as filterable categories, the JSON-LD schema in the page head, and an Open Graph / Twitter card for social previews.

On publish, three things happen in parallel:

  • Sitemap entry added and submitted to Google & Bing
  • IndexNow ping sent so AI crawlers (Bing, Perplexity, OpenAI) discover the URL within minutes
  • Project added to the internal dashboard counts and GC relationship map

Why correct schema is non-negotiable

Schema.org structured data is the contract between your website and machine readers — search engines, AI assistants, and knowledge graphs.

schema.org
A

It's how AI search answers a question

Google SGE, Bing Copilot, Perplexity, and ChatGPT search don't crawl pages the way classic Google does. They synthesize answers from indexed facts. Schema gives them the typed facts to cite — "this contractor, this service, this location."

B

It survives copy changes

The body text on a project page may be rewritten. The schema block holds the canonical facts — GC, address, unit count, service category — so the page's meaning doesn't drift even when the prose does.

C

Wrong schema is worse than none

A Service with a missing areaServed, or a LocalBusiness with the wrong NAP, can get the entire site disqualified from rich results. Validation has to be part of the pipeline, not an afterthought.

D

It links pages into a graph

Every project page references the same LocalBusiness entity for J Becher. AI assistants follow those references to build a confident picture: 758 permits, 33 counties, IBEW Local 292, multifamily strength.

E

It's how you show up in zero-click answers

The future of search isn't a list of blue links — it's an AI answer with two or three cited contractors. Without schema, you're not in the answer. With schema, you're a candidate every time the question matches your service area.

F

It compounds across 758 pages

A single project page with correct schema is a data point. 758 project pages with consistent schema is a market signal. Both Google and the AI assistants pick up on the density and breadth of structured coverage.

The compounding effect

Every additional enriched project page strengthens the entity graph: same business, more services, broader area served. AI ranks confident entities ahead of fuzzy ones.
